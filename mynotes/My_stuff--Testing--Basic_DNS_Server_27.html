Basic DNS Server


Great question this is exactly the kind of low-level testing that gives you real insight into DNS behavior. From your Ubuntu terminal, you can test nameservers in several different ways depending on what you want to verify (reachability, correctness, authority, performance, etc.).

I'll walk you through the most useful tests and what each one tells you.

  1. 1.

    1. Basic DNS query (is the nameserver working?)

Use dig to query a specific nameserver directly:

dig @ns.wedos.com example.com

dig @ns1.clouvy.net example.com

dig @prime.clouvy.net example.com

What this tests:

  • Whether the nameserver responds

  • Whether it returns valid DNS data

  1. 2.

    2. Query a specific record type

For more targeted testing:

dig @ns.wedos.com example.com A

dig @ns1.clouvy.net example.com MX

dig @prime.clouvy.net example.com NS

Useful for:

  • Verifying zone content

  • Checking if records are consistent across NS

  1. 3.

    3. Check authoritative response

dig @ns.wedos.com example.com +norecurse

Look for:

  • aa flag (Authoritative Answer)

If present this server is authoritative for the domain

  1. 4.

    4. Compare multiple nameservers (VERY important)

dig @ns.wedos.com example.com A +short

dig @ns1.clouvy.net example.com A +short

dig @prime.clouvy.net example.com A +short

You want:

  • Identical results across all NS

If not zone is not synchronized

  1. 5.

    5. Measure response time

dig @ns.wedos.com example.com | grep "Query time"

Helps you compare:

  • Performance of each NS

  • Latency differences

  1. 6.

    6. Test recursion (important for security)

dig @ns.wedos.com google.com

If it returns an answer:

  • recursion is enabled

If you see:

WARNING: recursion requested but not available

  • ◇ ◇ good authoritative-only server

  1. 7.

    7. Check NS records from the root perspective

dig example.com NS

Shows:

  • Which nameservers are officially delegated

Then verify each one manually.

  1. 8.

    8. Trace full DNS resolution path

dig +trace example.com

Shows:

  • Root TLD authoritative NS chain

Very useful for debugging delegation issues.

  1. 9.

    9. Query using specific source IP (advanced testing)

If you want to simulate different resolvers:

dig example.com @ns.wedos.com -b YOUR_IP

Useful when:

  • debugging geo-based DNS

  • testing ACL rules

  1. 10.

    10. Test zone transfer (AXFR)

dig @ns.wedos.com example.com AXFR

If it works:

  • zone transfer is allowed

If it fails:

  • expected for public NS (security)

  1. 11.

    11. Use

    nslookup

    (simpler but less powerful)

nslookup example.com ns.wedos.com

Good for quick checks, but dig is preferred.

  1. 12.

    12. Check if nameserver is reachable (network level)

ping ns.wedos.com

or better:

nc -zv ns.wedos.com 53

Tests:

  • basic connectivity

  • port 53 open

Pro tip (very useful in your WEDOS context)

If you're debugging customer issues, this combo is gold:

for ns in ns.wedos.com ns1.clouvy.net prime.clouvy.net; do

echo "=== $ns ==="

dig @$ns example.com A +short

done

Quickly shows inconsistencies across nameservers.
