Great — these two edge cases come up surprisingly often with reverse proxy services like WGP, especially when customers use automatic TLS (Let’s Encrypt) and modern protocols like HTTP/2 or HTTP/3. Understanding them will make troubleshooting much faster.

1️⃣ Edge Case: Let’s Encrypt / ACME validation fails through WGP

This happens when the customer tries to issue or renew a TLS certificate while traffic is routed through WGP.

There are two common ACME validation methods:

HTTP-01 challenge

Let's Encrypt requests:

http://domain.tld/.well-known/acme-challenge/<token>

The validation server expects the origin to return a specific token.

What normally happens without a proxy

Let's Encrypt

↓

domain.tld

↓

origin server

↓

/.well-known/acme-challenge/token

Origin returns the token → validation succeeds.

What happens with WGP in front

Let's Encrypt

↓

domain.tld

↓

WGP proxy

↓

origin server

This usually works, but can break if:

Problem 1 — WGP blocks the request

Security filtering may treat the validation bot as suspicious traffic.

Possible causes:

  • rate limiting

  • bot protection

  • firewall rules

  • country blocking

Problem 2 — redirect logic interferes

Example:

http://domain.tld

  • redirect → https://www.domain.tld

Now the ACME server follows multiple redirects before reaching the token.

Some ACME implementations tolerate this poorly.

Problem 3 — the hostname isn’t protected

Example configuration:

www.domain.tld → WGP

domain.tld → origin

If the certificate is requested for:

domain.tld

but only www goes through WGP, validation may fail depending on redirect logic.

How WGP mitigates this

That DNS record your colleague mentioned:

_acme-challenge.www CNAME _acme-challenge.domain.tld.wedos.global

This is DNS-01 validation delegation.

Flow:

Let's Encrypt

↓

check TXT record

_acme-challenge.www.domain.tld

↓

CNAME

↓

_wedos.global validation service

WEDOS automatically places the correct TXT record there.

Result:

✔ validation works

✔ customer doesn’t need to manually manage TXT records

✔ WGP can automate certificate issuance

2️⃣ Edge Case: HTTP/2 or HTTP/3 ALPN issues during redirects

This one is more subtle but happens in real deployments.

ALPN

When a TLS connection starts, the client and server negotiate the protocol:

Example:

ALPN: h2, http/1.1

Server chooses:

h2

for HTTP/2.

What happens in a proxy chain

The browser connects to:

User → WGP

Protocol negotiation happens here.

Example:

Browser → WGP : HTTP/2

But then WGP connects to origin using:

WGP → origin : HTTP/1.1

This is completely normal.

Where problems appear

Redirect chains can behave differently depending on protocol.

Example:

http://domain.tld

  • redirect → https://domain.tld
  • redirect → https://www.domain.tld

With HTTP/2 multiplexing, the browser may reuse the connection.

Some servers incorrectly handle:

:authority header

Host header

redirect location

This can create strange behaviors like:

  • infinite redirect loops

  • 421 Misdirected Request

  • mixed hostnames

Example real-world issue

Browser request:

https://domain.tld

Connection negotiated:

HTTP/2

Origin expects:

www.domain.tld

But proxy forwarding logic sends:

Host: domain.tld

Origin returns redirect.

Browser reuses connection.

Server responds with:

421 Misdirected Request

because HTTP/2 connection is tied to the wrong hostname.

3️⃣ Why these problems appear with reverse proxies

Because the architecture becomes:

Browser

↓

WGP edge proxy

↓

Origin server

Each layer may:

  • terminate TLS

  • rewrite headers

  • modify redirects

  • change protocol versions

Small configuration mistakes can cause subtle failures.

4️⃣ A very useful WGP troubleshooting workflow

When a customer says:

“My website broke after enabling WGP.”

Run these five checks.

Step 1 — Check DNS

dig domain.tld

dig www.domain.tld

Verify both resolve to WGP.

Step 2 — Test origin directly

curl -I https://origin-ip

Confirm origin works without WGP.

Step 3 — Test via WGP

curl -I https://domain.tld

curl -I https://www.domain.tld

Check redirects and headers.

Step 4 — Inspect redirect chain

curl -IL https://domain.tld

Look for:

301

302

loops

hostname mismatches

Step 5 — Check TLS / ALPN

curl -v https://domain.tld

Look for:

ALPN

certificate names

protocol

5️⃣ One final insight that will help you in your role

There are three completely separate canonical layers:

DNS canonical

CNAME

HTTP canonical

301/302 redirects

TLS canonical

certificate SAN names

These three layers must agree.

If any of them disagree, the site can break in confusing ways.
