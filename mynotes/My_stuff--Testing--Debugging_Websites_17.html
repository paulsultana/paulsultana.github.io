Debugging Websites


CURL for testing IP using Host Name

curl -kv -H "Host: tublat.com" https://51.89.97.16/

This command is manually connecting to a specific IP address over HTTPS, while pretending to request the domain tublat.com.

🔍 What Each Part Does

curl

Command-line HTTP client used to send requests and inspect responses.

-k

Insecure mode — disables TLS certificate verification.

Why this matters:

  • You are connecting to https://51.89.97.16
  • The TLS certificate on that server is almost certainly issued for tublat.com, NOT for 51.89.97.16
  • This would normally cause a certificate validation error
  • -k tells curl: “Ignore certificate mismatch or trust issues.”

Without -k, curl would likely fail with:

SSL: certificate subject name does not match target host name

-v

Verbose mode.

Shows:

  • TLS handshake
  • ALPN negotiation (HTTP/1.1, HTTP/2)
  • Certificate details
  • Request headers sent
  • Response headers received
  • Connection details

This is extremely useful when debugging:

  • WGP routing
  • Origin connectivity
  • Certificate issues
  • HTTP/2 vs HTTP/1.1 behavior

-H "Host: tublat.com"

Manually sets the HTTP Host header.

This is the most important part.

Even though you are connecting to:

https://51.89.97.16/

The HTTP request will contain:

Host: tublat.com

Why this matters

Web servers use the Host header for:

  • Virtual hosting (multiple domains on one IP)
  • Routing to correct backend
  • CDN/WAF domain matching

So this command says:

“Connect to IP 51.89.97.16, but treat the request as if I’m requesting tublat.com.”

https://51.89.97.16/

You are connecting directly to that IP over port 443 (HTTPS).

This bypasses:

  • DNS resolution
  • Anycast DNS routing
  • CDN hostname resolution

It does NOT bypass:

  • Reverse proxies running on that IP
  • WAF rules
  • Virtual host configuration

🧠 What This Is Typically Used For

In WGP / reverse proxy / DNS troubleshooting, this command is used to:

1️⃣ Test the origin server directly (bypass DNS)

If 51.89.97.16 is the origin server IP, this checks:

  • Is the origin alive?
  • Is it responding correctly?
  • Is it serving the correct virtual host?
  • Does it return 200, 500, 503?

2️⃣ Check if WGP is in front or not

If 51.89.97.16 is:

  • A WGP proxy IP → you’re still hitting WGP
  • A backend/origin server → you’re bypassing WGP

You can tell by headers in verbose output:

  • x-protected-by
  • x-request-id
  • WEDOS headers
  • Or pure nginx/apache headers

3️⃣ Test virtual host configuration

If the server hosts multiple domains:

Without -H "Host: tublat.com"

  • You might get the default vhost (wrong site)

With it

  • You get the specific tublat.com configuration

⚠ Important TLS Detail (Subtle but Advanced)

Even though you set:

-H "Host: tublat.com"

TLS SNI (Server Name Indication) is still based on the URL hostname (51.89.97.16).

That means:

  • TLS handshake uses SNI = 51.89.97.16
  • HTTP request uses Host = tublat.com

This mismatch is why -k is required.

If you wanted full SNI + Host alignment while connecting to an IP, you'd use:

curl -kv --resolve tublat.com:443:51.89.97.16 https://tublat.com/

That is cleaner for HTTPS debugging because:

  • SNI = tublat.com
  • Host header = tublat.com
  • Connection goes to 51.89.97.16

🎯 In a WGP Context

This command helps answer:

  • Is the origin working without WGP?
  • Is WGP the problem?
  • Is DNS misconfigured?
  • Is the backend IP correct?
  • Is the certificate correct?
  • Is the virtual host configured properly?

🧩 Summary

This command:

curl -kv -H "Host: tublat.com" https://51.89.97.16/

Means:

“Connect directly to IP 51.89.97.16 over HTTPS, ignore TLS certificate errors, and request the virtual host for tublat.com while showing full debugging output.”

It is primarily used to:

  • Bypass DNS
  • Test origin servers
  • Debug reverse proxy/WGP routing
  • Validate backend configuration
