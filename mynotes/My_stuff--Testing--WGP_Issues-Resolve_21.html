Step 1 — Test each WGP proxy IP directly

Run these two commands:

curl -I --resolve www.auma.de:443:185.8.237.5 https://www.auma.de

curl -I --resolve www.auma.de:443:185.8.237.6 https://www.auma.de

This forces curl to connect to one specific proxy node.

Explanation:

--resolve hostname:port:IP

So curl will:

connect to 185.8.237.5

send Host: www.auma.de

use TLS SNI www.auma.de

That simulates a real client request but lets you choose the proxy node.

Step 2 — Run a continuous test

Now run this loop:

while true; do

date

echo "Node 185.8.237.5"

curl -s -o /dev/null -w "%{http_code}\n" --resolve www.auma.de:443:185.8.237.5 https://www.auma.de

echo "Node 185.8.237.6"

curl -s -o /dev/null -w "%{http_code}\n" --resolve www.auma.de:443:185.8.237.6 https://www.auma.de

echo

sleep 5

done

Let it run for several minutes.

What the results tell you

Scenario A — both nodes return 503 at the same time

Example:

Node 185.8.237.5

503

Node 185.8.237.6

503

This strongly indicates:

WGP cannot reach the origin server

So the issue is likely:

  • origin firewall
  • origin connection limit
  • origin overload
  • upstream routing problem

Scenario B — only one node returns 503

Example:

Node 185.8.237.5

200

Node 185.8.237.6

503

This means:

one WGP proxy node cannot reach the backend

Then the WGP team needs to check that node.

Scenario C — everything returns 200

Example:

Node 185.8.237.5

200

Node 185.8.237.6

200

Then the issue may be:

  • geographic routing problem
  • monitoring node path issue
  • temporary failure earlier
  • origin firewall triggered only by certain traffic

Step 3 — capture headers during a failure

If you see a 503, immediately run:

curl -v --resolve www.auma.de:443:185.8.237.5 https://www.auma.de

(or .6 depending which node failed)

This will show headers like:

server:

via:

x-cache:

Those often reveal whether the error came from WGP or the origin.

Continuous test with conditional alert

while true; do

date

for ip in 185.8.237.5 185.8.237.6; do

code=$(curl -s -o /dev/null -w "%{http_code}" \

--resolve www.auma.de:443:$ip https://www.auma.de)

if [ "$code" -ge 200 ] && [ "$code" -lt 300 ]; then

echo "Node $ip → OK ($code)"

else

echo "Node $ip → ERROR ($code)"

fi

done

echo

sleep 5

done

Example output

Normal case:

Tue Mar 12 14:10:01 CET 2026

Node 185.8.237.5 → OK (200)

Node 185.8.237.6 → OK (200)

Failure case:

Tue Mar 12 14:10:06 CET 2026

Node 185.8.237.5 → ERROR (503)

Node 185.8.237.6 → OK (200)

This makes it very easy to see when something goes wrong.

Even better: add colors (very useful during long tests)

while true; do

date

for ip in 185.8.237.5 185.8.237.6; do

code=$(curl -s -o /dev/null -w "%{http_code}" \

--resolve www.auma.de:443:$ip https://www.auma.de)

if [ "$code" -ge 200 ] && [ "$code" -lt 300 ]; then

echo -e "\e[32mNode $ip → OK ($code)\e[0m"

else

echo -e "\e[31mNode $ip → ERROR ($code)\e[0m"

fi

done

echo

sleep 5

done

Green = success

Red = error

This makes intermittent 503s very obvious.

One more extremely useful tweak (optional)

You can also log only errors to a file for later analysis:

if [ "$code" -lt 200 ] || [ "$code" -ge 300 ]; then

echo "$(date) Node $ip ERROR ($code)" >> errors.log

fi

That way if a failure happens overnight you have a record.
